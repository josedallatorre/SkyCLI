\section{Setup}
This section describes how to correctly set up the operating system on the Raspberry Pi, configure the repository, and install the required libraries for the proposed solution. \\
After installing the Raspberry Pi OS following \href{https://www.raspberrypi.com/documentation/computers/getting-started.html}{official guide}, it is necessary to enable SPI (Serial Peripheral Interface). This can be done by running:
\begin{minted}{bash}
sudo raspi-config
\end{minted}
Then navigate to "Interface" then "SPI" and enable it. \\
Next, install the WiringPi library, which allows high-performance access to GPIO:
\begin{minted}[breaklines]{bash}
wget https://github.com/WiringPi/WiringPi/releases/download/3.16/wiringpi_3.16_armhf.deb
sudo dpkg -i wiringpi_3.16_armhf.deb
\end{minted}
Two files inside the LoRa-RaspberryPi folder must be modified:
\begin{itemize}
        \item Makefile: ensure \#PYTHONVER=python3.11 matches the system’s Python version
        \item lora.c: add the following line at the top for compatibility with newer Python versions: \#define PY\_SSIZE\_T\_CLEAN 
\end{itemize}
To check the installed Python version:
\begin{minted}{bash}
python3 --version
\end{minted}
These modifications have already been applied in the \href{https://github.com/josedallatorre/SkyCLI/}{updated repository} (see the src folder). \\
Finally, compile the project using:
\begin{minted}{bash}
make all
\end{minted}
\section{Proposed solution}
This section presents the proposed solution. It begins with an overview of the folder structure, followed by a description of the sender and receiver components. Finally, the data analysis process is discussed.
\subsection{Folder structure}
Inside the repository there are three folder:
\begin{itemize}
        \item report: this folder can be ignored since it has been used to store the files needed for this report.
        \item src: inside this folder there's the code for the receiver and the sender, then the libraries used by both. There's also a utils folder with helper functions for logging and converting digits.
        \item analysis: inside this folder there's a notebook with the code for the analysis that will be explained later.
\end{itemize}
\subsection{Sender}
Inside the sender folder, there are two files:
hf-sender.py: initially used to test the maximum message throughput. However, it was not used in the final experiments due to reliability issues. \\
During testing, it was observed that messages were often corrupted or not correctly received. \\
After reviewing \href{https://www.thethingsnetwork.org/docs/lorawan/spreading-factors/}{relevant documentation}, the implementation was revised in a new file, sender.py. In this version, message transmission is delayed based on airtime. \\
Airtime is computed using the formula defined in \href{https://www.mouser.com/pdfdocs/semtech-lora-modem-design.pdf}{Semtech's LoRa Modem Designer's Guide (AN1200.13)}. Additionally, an \href{https://avbentem.github.io/airtime-calculator/ttn/eu868}{online tool} was used to visualize how airtime is affected by parameters such as payload size and data rate. \\
To start sending messages use the following command inside the "src" folder:
\begin{minted}[breaklines]{bash}
python -m sender.sender 
\end{minted}
\subsection{Receiver}
Insider the receiver folder there are two files:
receiver.py: used to verify message reception and logging. \\
Initial tests showed that iterating over a fixed number of messages was ineffective due to synchronization issues between sender and receiver. \\
Therefore, a time-based approach was adopted: messages are sent and received over a fixed duration (2 minutes). \\
To start the receiver use the following command inside the "src" folder:
\begin{minted}[breaklines]{bash}
python -m receiver.hf-receiver 
\end{minted}
\subsection{Utils}
Inside the utils folder there are two files:
\begin{itemize}
        \item conversion.py: contains two helper function that convert a positive number to its digit representation in base b.
        \item logging.py: contains one helper function that create a logger and setup the folders to save the data from the communicator.
\end{itemize}
The first file has been created to solve the following problem: since the LoRaRaspberryPi library only transmit bytes of data it's necessary to convert the counter to a list of digits in base 255. \\
It's has been used a list of digits to be able to send a counter greater than 255.
\section{Analysis}
In this section will be presented the data analysis based on the tests made outside the SMACT 3 laboratory. \\
In total 7 test has been done:
\begin{itemize}
        \item  first test: communication in the parking area of the laboratory, $\sim$100 meters. LoRa spreading factor set to 7.
        \item  second test: same as the first test but with a spreading factor set to 12.
        \item  third test: from the water storage outside the canteen to the trees, $\sim$100 meters, LoRa spreading factor set to 12.
        \item  fourth test: same as the third test but with more obastacles (trees) between the RaspberryPi.
        \item  fifth test: from the tree next to the canteen to "Canale Ledra". LoRa spreading factor set to 12.
        \item  sixth test: communication from Autonomous System laboratory to Cybersecurity laboratory. LoRa spreading factor set to 10.
        \item  seventh test: communication inside the Autonomous System laboratory. LoRa spreading factor set to 10.
\end{itemize}
\subsection{Data Cleaning}
The analysis starts with the installation of the required packages and the definition of the function already presented before. The collected data was preprocessed to remove inconsistencies. \\
This step revealed:
\begin{itemize}
        \item 5 errors in the sender logs.
        \item 1 invalid message in the receiver logs.
\end{itemize}
Given the small proportion of errors relative to the total number of transmitted messages, their impact on the overall results is minimal. Therefore, they are not expected to significantly affect the validity of the analysis. \\

\subsection{Latency Analysis}
The latency analysis was conducted by matching each received message with its corresponding transmitted message using the message counter. This allowed the computation of end-to-end transmission delay for each successfully received packet. \\
Negative latencies and unmatched messages were filtered out to ensure data consistency.
In particular, the following metrics were considered:
\begin{itemize}
        \item Median latency (typical performance).
        \item Maximum latency (worst-case scenario)
        \item 99th percentile latency (tail behavior)
\end{itemize}
These metrics allow the identification of latency spikes and help define acceptable performance bounds.

\section{Conclusion}
This report evaluated the performance of a LoRa-based communication system under different environmental conditions and configurations. \\
The experimental results demonstrate that the system achieves high reliability, with only a minimal number of transmission errors observed across all tests. This confirms that LoRa is well-suited for long-range, low-power communication even in the presence of obstacles. \\
Latency analysis revealed a clear trade-off between reliability and performance. Higher spreading factors improve message delivery success but introduce increased latency and variability. This trade-off is a fundamental characteristic of LoRa communication and must be considered when designing real-world applications. \\
Environmental conditions also play a significant role. Indoor scenarios provide more stable performance, while outdoor environments introduce greater variability due to obstacles and interference. \\
Overall, the system performs reliably with acceptable latency, making it suitable for low-throughput IoT applications such as monitoring and telemetry. \\
Future work could focus on:
\begin{itemize}
    \item Automating synchronization between sender and receiver
    \item Testing over longer distances
    \item Evaluating performance under higher network load
\end{itemize}
